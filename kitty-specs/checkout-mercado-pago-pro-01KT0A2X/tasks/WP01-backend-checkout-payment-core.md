# WP01 - Backend Checkout And Payment Core

## Objective

Implement the backend checkout workflow: create payment preferences, validate
provider notifications, update order state, record delivery/consent fields, send
confirmation emails, and expire processing orders via a management command.

## Context

- Spec: `kitty-specs/checkout-mercado-pago-pro-01KT0A2X/spec.md`
- Plan: `kitty-specs/checkout-mercado-pago-pro-01KT0A2X/plan.md`

## Subtasks

### T001 - Create checkout app skeleton and wire routes

- Create the new checkout app module with the required Django scaffolding.
- Add URLs for the payment initiation and webhook endpoints.
- Register the app in settings and include the checkout URLs in the API routing.

### T002 - Extend Order model with delivery and LGPD consent fields

- Add delivery address fields and LGPD consent fields as described in
  `data-model.md`.
- Add a provider status field for the last known payment status.
- Create and register the migration, ensuring existing data remains valid.

### T003 - Implement payment initiation endpoint and preference creation

- Validate that the authenticated customer owns the order and that address
  fields and LGPD consent are present.
- Create a Mercado Pago preference and store the preference id on the order.
- Transition the order status to processing inside an atomic transaction.
- Return the provider `init_point` URL and preference id per the contract.

### T004 - Implement webhook verification and order status updates

- Validate the provider signature header before processing updates.
- Fetch the payment status from the provider and map it to order state updates.
- Keep pending payments in processing; approve and reject as specified.
- Ensure updates are idempotent and safe to replay.

### T005 - Send confirmation email on paid orders

- Trigger a confirmation email when the order transitions to paid.
- Ensure the webhook response is not blocked by email delivery.

### T006 - Add expiration management command for processing orders

- Implement a management command that cancels processing orders older than the
  configured expiration window (default 48 hours).
- Provide a clear summary of how many orders were expired.

## Validation

- Manual API smoke test for payment initiation and webhook update.
- Run the expiration command against a test order and verify status changes.

## Definition Of Done

- Payment initiation and webhook endpoints behave per the contract.
- Order state transitions follow the spec, including pending expiration.
- Delivery and consent fields are stored on the order.
- Confirmation email triggers on paid orders.
- Expiration command cancels stale processing orders.

## Risks And Notes

- Ensure signature validation rejects unsigned notifications.
- Keep status updates atomic to avoid partial state transitions.
