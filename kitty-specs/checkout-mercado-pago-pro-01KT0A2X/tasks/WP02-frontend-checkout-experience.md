# WP02 - Frontend Checkout Experience

## Objective

Deliver the checkout UI flow: address entry with CEP auto-fill, LGPD consent
confirmation, payment redirect, and success/failure outcomes with order
summaries and retry options.

## Context

- Spec: `kitty-specs/checkout-mercado-pago-pro-01KT0A2X/spec.md`
- Plan: `kitty-specs/checkout-mercado-pago-pro-01KT0A2X/plan.md`

## Subtasks

### T007 - Build checkout address form with CEP auto-fill and validation

- Extend `/checkout/[id]` to capture delivery address fields.
- On valid CEP input, fetch ViaCEP data and populate fields while allowing edits.
- Validate required fields before enabling payment.

### T008 - Add LGPD consent gating and payment redirect

- Add the consent checkbox and link to the privacy policy.
- Disable the payment button until consent is checked and the form is valid.
- Call the payment initiation endpoint and redirect to the returned `init_point`.

### T009 - Implement success and failure pages with order summary and retry

- Build `/checkout/success` to show the order summary.
- Build `/checkout/failure` with a retry option back to the checkout flow.
- Ensure both pages handle missing or invalid order ids gracefully.

### T010 - Show "Processando" status in order list and detail views

- Update status mapping so processing orders show "Processando".
- Confirm list and detail routes use the same label mapping.

### T011 - Update frontend services/types for checkout endpoints

- Add API client helpers and types for the checkout payment endpoint.
- Align order and checkout types with new delivery/consent fields as needed.

## Validation

- Walk through the checkout flow locally and confirm redirection and return
  screens display as expected.
- Validate that the payment button stays disabled until the form is valid and
  consented.

## Definition Of Done

- Checkout UI collects address and consent, then redirects to the provider.
- Success and failure pages render with appropriate actions.
- Order status labels show processing state as "Processando".
- Frontend uses the shared API client for checkout calls.

## Risks And Notes

- Preserve existing checkout behavior for authenticated customers.
- Avoid blocking the UI if ViaCEP fails; allow manual entry.
